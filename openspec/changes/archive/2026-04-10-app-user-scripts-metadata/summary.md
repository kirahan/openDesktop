# app-user-scripts-metadata

## 概述

**一句话**：为每个已注册应用提供 **用户脚本** 的 **持久化** 与 **UserScript 元数据解析**，**完整保留 `@match` 等字段**；**本阶段不实现** 任何基于 `@match` 的 URL 匹配、过滤或自动注入，并在设计中标明 **SPA 客户端路由** 与未来 URL 匹配之间的张力。

**背景与动机**：用户希望以油猴风格管理脚本，且脚本 **相对 app** 归属即可缩小信任域；若一次性实现「匹配 + 新文档注入」，易与 **SPA 无整页导航** 场景冲突。先落地 **存储 + 解析契约**，避免半成品 match 行为误导用户。

**影响范围**：`packages/core`（解析模块、按 app 存储、HTTP CRUD、测试）；可选文档；**不包含** Studio 完整 UI（可后续变更）。

## 设计方案

### 改动点

| # | 改动 | 说明 | 涉及文件（预期） |
|---|------|------|-------------------|
| 1 | UserScript 解析器 | 抽取元数据与 `matches[]`，`@grant` 仅 `none` | `packages/core/src/...` 新建模块 |
| 2 | 按 app 持久化 | JsonStore 或等价，`appId` 键控 | `packages/core/src/store/` 或会话数据旁 |
| 3 | HTTP API | CRUD + 列表，返回含 `matches` 的结构 | `createApp.ts` 或路由注册模块 |
| 4 | 测试 | 解析单测 + HTTP 集成测 | `*.spec.ts` / `*.http.test.ts` |

### 关键决策

| 决策 | 选择 | 为什么不选其他方案 |
|------|------|-------------------|
| `@match` 本阶段 | 只存不用 | 避免 SPA 下错误语义；与「先契约后执行」一致 |
| `@grant` | 仅 `none` | 与不做 GM 兼容一致 |
| 存储键 | 按 `appId` | 用户心智「脚本属于应用」 |

### 技术细节

- 源文件 **全文**持久化，解析结果可缓存或按需重算。
- **不**调用 CDP、**不**根据 URL 过滤脚本列表（除非 API 另有按 id 查询）。
- SPA：见 `design.md` 第五节——未来若做自动注入，需额外策略，非本变更范围。

## 验收标准

> 测试与产品验收以本节为准（摘自 `specs/app-user-scripts/spec.md`）。

| # | 场景 | 操作 | 预期结果 |
|---|------|------|----------|
| 1 | 按 app 持久化 | 为某 `appId` 创建含 UserScript 头的脚本并列表 | 能列出且归属正确 |
| 2 | 多条 @match 入库 | 源文含多条 `@match` | 结构化元数据含全部字符串 |
| 3 | 缺必填 | 缺少 `@name`（若定义为必填）等 | 创建失败，无脏数据 |
| 4 | grant 非法 | `@grant` 非 `none` | 拒绝保存 |
| 5 | 不匹配不注入 | 任意 URL / 导航 | 不因 `@match` 自动执行脚本 |
| 6 | SPA 说明 | 打开 `design.md` | 存在 SPA 与 `@match` 张力说明 |

## 实现清单

| # | 任务 | 说明 | 依赖 |
|---|------|------|------|
| 1 | 解析器 | UserScript 块 + `matches[]` + grant 校验 | — |
| 2 | 存储 | 按 app CRUD | 1 |
| 3 | HTTP API | 鉴权、返回 metadata | 1–2 |
| 4 | 测试与文档 | 单测、集成测、README 提示 | 3 |

## 测试要点

- 多条 `@match`、非法 `@grant`、缺 `@name`。
- **负向**：无 API 仅因 URL 触发脚本执行（若难以测「无」，则测创建后无后台注入任务）。
- 回归：现有会话/CDP 行为无破坏。
